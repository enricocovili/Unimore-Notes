# 1. Introduzione e Analisi dei requisiti

### 1.1 Contesto

Il calcolo ad alte prestazioni (HPC) su GPU è una risorsa costosa e scarsa. In un ambiente distribuito (es. cluster universitario o aziendale), molte macchine dispongono di GPU sottoutilizzate mentre altre sono sovraccariche.
L'obiettivo del progetto è sviluppare un **Middleware di Load Balancing** che distribuisca dinamicamente i task di calcolo (es. training di reti neurali, rendering) verso i nodi (Worker) più scarichi.

### 1.2 Requisiti Funzionali

* **Registrazione Nodi:** I nodi Worker devono potersi registrare al sistema centrale dichiarando le specifiche della propria GPU (VRAM, CUDA Cores).
* **Sottomissione Job:** I client inviano task di calcolo.
* **Distribuzione (Balancing):** Il sistema deve assegnare il job al nodo migliore secondo un algoritmo configurabile.
* **Monitoraggio:** Il sistema deve conoscere lo stato (Idle, Busy, Offline) di ogni nodo in tempo reale.

### 1.3 Requisiti Non Funzionali

* **Scalabilità:** Il sistema deve supportare l'aggiunta dinamica di nuovi nodi.
* **Affidabilità:** Se un nodo cade, il task deve essere riassegnato.

# 3. Architettura Software e Design Patterns

Questa è la parte centrale della tesina. Per risolvere i problemi di scalabilità e manutenibilità, sono stati applicati i seguenti pattern.

### 3.1 Strategy Pattern (Per l'Algoritmo di Bilanciamento)

**Problema:** Non esiste un unico modo perfetto per bilanciare il carico. A volte serve il "Round Robin", altre volte il "Least Connections" (chi ha meno job) o "Hardware Capability" (chi ha più VRAM libera).
**Soluzione:** Usiamo il pattern **Strategy** per rendere l'algoritmo di bilanciamento intercambiabile a runtime senza modificare la classe `LoadBalancer`.

**Struttura delle Classi (Diagramma delle Classi parziale):**

* `Context`: **LoadBalancer**
* `Strategy Interface`: **IBalancingStrategy**
* `Concrete Strategies`: **RoundRobinStrategy**, **LeastVramUsageStrategy**.

**Implementazione (Pseudo-codice/Java):**

```java
// Interfaccia comune
public interface IBalancingStrategy {
    WorkerNode selectNode(List<WorkerNode> nodes, Job job);
}

// Strategia 1: Sceglie chi ha più VRAM libera
public class LeastVramUsageStrategy implements IBalancingStrategy {
    public WorkerNode selectNode(List<WorkerNode> nodes, Job job) {
        // Logica di ordinamento per VRAM decrescente
        return nodes.stream()
            .max(Comparator.comparing(WorkerNode::getFreeVram))
            .orElse(null);
    }
}

// Context
public class LoadBalancer {
    private IBalancingStrategy strategy;

    public void setStrategy(IBalancingStrategy strategy) {
        this.strategy = strategy;
    }

    public void dispatch(Job job) {
        WorkerNode target = strategy.selectNode(availableNodes, job);
        target.execute(job);
    }
}

```

### 3.2 Observer Pattern (Per il Monitoraggio dei Nodi)

**Problema:** Il `LoadBalancer` deve sapere immediatamente quando un Worker finisce un lavoro o se va offline (surriscaldamento GPU), per non inviargli altri task. Un polling continuo ("sei libero? sei libero?") è inefficiente.
**Soluzione:** Usiamo l'**Observer Pattern**. I Worker sono i "Soggetti" e il Balancer è l'"Osservatore". Quando lo stato di un Worker cambia, notifica il Balancer.

**Struttura:**

* `Subject`: **WorkerNode**
* `Observer`: **INodeStatusListener** (implementato da `LoadBalancer`)

**Implementazione:**

```java
// Observer Interface
public interface INodeStatusListener {
    void onStatusChange(WorkerNode node, NodeStatus newStatus);
}

// Concrete Observer
public class LoadBalancer implements INodeStatusListener {
    @Override
    public void onStatusChange(WorkerNode node, NodeStatus newStatus) {
        if (newStatus == NodeStatus.IDLE) {
            System.out.println("Nodo " + node.getId() + " è libero. Assegno coda.");
            assignNextJob(node);
        } else if (newStatus == NodeStatus.OFFLINE) {
            removeFromPool(node);
        }
    }
}

// Subject
public class WorkerNode {
    private List<INodeStatusListener> listeners = new ArrayList<>();
    private NodeStatus status;

    public void setStatus(NodeStatus status) {
        this.status = status;
        notifyListeners(); // Notifica il cambiamento
    }
    
    // ... metodi attach/detach
}

```

### 3.3 Singleton Pattern (Per la Configurazione e Logging)

**Problema:** Esiste una sola configurazione globale del sistema (es. porta di ascolto, timeout, soglia massima temperatura GPU). Avere più istanze della configurazione caricate in memoria può creare inconsistenze.
**Soluzione:** Utilizzare il **Singleton** per la classe `SystemConfiguration`.

```java
public class SystemConfiguration {
    private static SystemConfiguration instance;
    private int maxGpuTemp;

    private SystemConfiguration() {
        // Carica impostazioni da file
        this.maxGpuTemp = 85; 
    }

    public static synchronized SystemConfiguration getInstance() {
        if (instance == null) {
            instance = new SystemConfiguration();
        }
        return instance;
    }
}

```

# 4. Dettagli Implementativi

### 4.1 Tecnologie Scelte

* **Linguaggio:** Java. La scelta potrebbe cadere su un qualsiasi altro linguaggio di programmazione, in quanto non siamo limitati da problemi di performance (il bottleneck è dato dalla comunicazione)
* **Comunicazione:** gRPC o REST API per la comunicazione tra Client, Balancer e Worker.
* **Librerie GPU:** Integrazione con NVIDIA NVML.

### 4.2 Complessità dell'Algoritmo

Se  è il numero di nodi worker attivi, la strategia *LeastVramUsage* richiede un ordinamento o una ricerca del massimo, portando a una complessità computazionale per ogni richiesta di:
$O(n \log n)$ (ricerca binaria)

# 5. Conclusioni e Sviluppi Futuri

Il sistema progettato offre un'architettura modulare grazie all'uso dei Design Pattern. L'uso dello **Strategy Pattern** permette di adattare il software a cluster eterogenei (es. bilanciare per potenza di calcolo invece che per memoria), mentre l'**Observer** garantisce reattività in tempo reale.

Sviluppi futuri potrebbero includere un modulo di **Machine Learning** per prevedere la durata dei job e ottimizzare ulteriormente lo scheduling (Predictive Load Balancing).
